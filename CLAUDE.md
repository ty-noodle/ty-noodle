## Commands

```bash
npm run dev          # Start dev server at http://localhost:3000
npm run build        # Production build
npm run start        # Serve production build locally
npm run lint         # ESLint (Next.js core-web-vitals + TypeScript)

```

Run `npm run gen:types` after modifying SQL in `supabase/migrations/`. Requires `SUPABASE_PROJECT_ID` and `SUPABASE_ACCESS_TOKEN` in `.env.local`. Never hand-edit `src/types/database.ts`.

Pre-PR checks: `npm run lint` + successful `npm run build` (no automated test suite yet).

## Architecture

**T&YNoodle** is a Next.js 16 App Router PWA for a noodle distribution business. UI is in Thai (Sarabun + IBM Plex Mono + Instrument Sans via Google Fonts; LINE Seed Sans TH local fonts in `src/app/fonts/` used in print layouts). Uses Supabase (PostgreSQL + RLS), LINE LIFF for mobile integration, and Tailwind CSS 4. React Compiler is enabled. Offline support via `/offline` page and service worker.

### Authentication

Custom PIN-based auth ‚Äî no Supabase Auth:
- User enters 6-digit PIN ‚Üí `src/app/login/actions.ts` calls `verifyPin()`
- PIN hashed with scrypt + pepper (`LOGIN_PIN_PEPPER`) and compared against `pin_lookup` table
- On success: HMAC-signed cookie (`tynoodle_session`) created via `src/lib/auth/session.ts`
- Session contains: user ID, display name, role (`admin | member`), org ID, expiry
- Middleware in `src/proxy.ts` protects `/dashboard/:path*` via `updateSession()` in `src/lib/supabase/middleware.ts`
- Use `requireAppSession()` from `src/lib/auth/authorization.ts` as an auth guard in server components/actions
- Use `requireAppRole("admin")` for admin-only routes (e.g. `/delivery`, `/orders/incoming`) ‚Äî redirects non-admins to `/dashboard`

### Supabase Clients

Four distinct clients ‚Äî use the right one:
- `src/lib/supabase/server.ts` ‚Üí `createClient()` ‚Äî for Server Components (read-only cookie access)
- `src/lib/supabase/action.ts` ‚Üí `createActionClient()` ‚Äî for Server Actions that need to write cookies
- `src/lib/supabase/browser.ts` ‚Äî for Client Components
- `src/lib/supabase/admin.ts` ‚Üí `getSupabaseAdmin()` ‚Äî service role, bypasses RLS (dashboard aggregates, admin ops)

### Data Fetching Pattern

Prefer Server Components for reads, Server Actions (`"use server"`) for mutations. Keep sensitive logic in `src/lib` with `"server-only"` imports to prevent client-side leakage.

### Domain Model

Key entities (all org-scoped via `organization_id`, RLS-enforced):
- **`organizations`** ‚Äî root tenant; `metadata` JSON stores org-level `address` and `phone` used in billing/delivery print headers
- **`customers`** ‚Äî has `customer_code` (TYS-prefix auto-seq), `billing_day`, `default_vehicle_id`, `line_user_id`
- **`products`** ‚Äî has `sku` (TYN-prefix auto-seq), `product_unit` enum (`kg|pack|bag|piece|tray`), `stock_quantity`, `reserved_quantity`
- **`product_sale_units`** ‚Äî multiple sale units per product (label, `base_unit_quantity`, cost override); the primary unit is on `products` itself
- **`product_images`** ‚Äî multiple images per product, sorted
- **`customer_product_prices`** ‚Äî per-customer sale prices keyed by `(customer_id, product_sale_unit_id)`
- **`orders`** / **`order_items`** ‚Äî status: `draft ‚Üí submitted ‚Üí confirmed ‚Üí cancelled`; order number format `TY-YYYYMMDD-XXXX`
- **`inventory_receipts`** / **`inventory_movements`** ‚Äî audit trail for stock; movement types: `receipt | reserve | issue | release | adjustment`
- **`vehicles`** ‚Äî delivery vehicles, org-scoped
- **`delivery_notes`** ‚Äî one per confirmed order; tracks dispatch status and item adjustments. Item quantities can be corrected post-confirmation via the `adjust_delivery_note_item(p_organization_id, p_delivery_note_item_id, p_new_quantity_delivered, p_adjusted_by, p_resolution_mode)` RPC ‚Äî current resolution mode used is `"lost"`
- **`billing_records`** ‚Äî locked billing statements generated from delivered orders; store a `snapshot_rows` JSON so past totals are immutable even if delivery notes are later edited

The order workboard (`/orders`) uses PostgreSQL RPC functions (`get_order_daily_store_summaries`, `get_order_daily_store_items`) defined in migrations ‚Äî modify those RPCs when changing workboard query logic. Billing sequence numbers are generated by the `next_billing_number(p_organization_id, p_billing_date)` RPC.

**Product sale unit cost modes**: each `product_sale_units` row has a `cost_mode` of `"derived"` (cost = base price √ó `base_unit_quantity`) or `"fixed"` (explicit `fixed_cost_price`). Use `getEffectiveSaleUnitCost()` from `src/lib/products/sale-unit-cost.ts` wherever sale unit cost is needed ‚Äî never reimplement this logic inline.

Key routes:
- `/order` ‚Äî LINE LIFF customer-facing order placement (no PIN auth, identified by `line_user_id`)
- `/orders` ‚Äî internal admin workboard (daily store summaries via RPC)
- `/orders/incoming` ‚Äî flat list of all incoming orders by date, admin-only
- `/orders/delivery-notes/[id]` ‚Äî printable delivery note for a confirmed order
- `/orders/delivery-notes/preview` ‚Äî preview/print combined delivery note
- `/orders/packing-list` ‚Äî printable product√óstore matrix for a given date (admin-only)
- `/delivery` ‚Äî delivery dispatch board (tick off delivered orders), admin-only
- `/delivery/print` ‚Äî printable delivery summary
- `/billing` ‚Äî billing statement creation + history, admin-only
- `/billing/print` ‚Äî preview/print a single billing statement; `?save=true` locks the snapshot
- `/dashboard` ‚Äî overview KPIs and quick links, middleware-protected
- `/dashboard/settings` ‚Äî admin-only settings under middleware protection (customers, products)
- `/stock` ‚Äî stock overview; `/stock/movements` ‚Äî movement history
- `/settings/customers` ‚Äî customer CRUD; `/settings/customers/pricing` ‚Äî per-customer sale price management
- `/settings/products` ‚Äî product catalog management
- `/settings/stock` ‚Äî stock receipts entry
- `/settings/vehicles` ‚Äî vehicle management
- `/api/geography` ‚Äî cascading province/district/subdistrict lookup from `thailand-geography.json` (used by address forms; responses are cached for 1 day)

### Route Protection

Middleware (`src/proxy.ts`) only runs on `/dashboard/:path*` and `/login`. All other routes (including `/order`, `/orders`, `/settings`, `/stock`) are **not** protected by middleware ‚Äî they rely on `requireAppSession()` called inside each server component/action individually.

### LINE Integration

Two distinct LINE integrations:
- **LIFF** (`NEXT_PUBLIC_LIFF_ID`): wraps the `/order` route via `LiffProvider` ‚Äî identifies customers by `line_user_id` from the LIFF context, no PIN required. Set `NEXT_PUBLIC_LIFF_MOCK=true` in `.env.local` to bypass real LINE auth during local dev ‚Äî **must be unset or `false` in production**. After order submission, the order receipt card is rasterised with `html2canvas` (dynamically imported) and offered as a shareable image.
- **Messaging API** (`LINE_CHANNEL_ACCESS_TOKEN` + `LINE_GROUP_ID`): `src/lib/line/notify.ts` pushes a Flex Message to a LINE group when a new order is submitted; both vars are optional ‚Äî notification silently skips if unset
- `/api/line/webhook` ‚Äî utility endpoint to capture `LINE_GROUP_ID` when the OA bot joins a group (logs it to console)

### Environment Variables

Required in `.env.local`:
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SUPABASE_PROJECT_ID` + `SUPABASE_ACCESS_TOKEN` ‚Äî needed for `npm run gen:types`
- `LOGIN_PIN_PEPPER` ‚Äî **never change after first deploy** (invalidates all PINs)
- `SESSION_SECRET` ‚Äî **never change after deploy** (invalidates all sessions)
- `NEXT_PUBLIC_LIFF_ID`

Optional:
- `LINE_CHANNEL_ACCESS_TOKEN` + `LINE_GROUP_ID` ‚Äî LINE group notifications (silently skipped if unset)
- `NEXT_PUBLIC_SITE_URL` ‚Äî notification deep-link base (defaults to `http://localhost:3000`)
- `NEXT_PUBLIC_LIFF_MOCK=true` ‚Äî local dev only, bypasses real LINE LIFF auth

See `.env.example` for the full template.

### Thai Date & Formatting Utilities

- **`fmtDateTH(iso)`** in `src/lib/utils/date.ts` ‚Äî converts `YYYY-MM-DD` ‚Üí `DD/MM/YYYY` with Buddhist Era year (+543). Use this everywhere dates are displayed in UI.
- **`fmtDateTimeTH(isoDateTime)`** ‚Äî same but includes `HH:mm` in Bangkok timezone (`Asia/Bangkok`).
- **`src/lib/format/baht-text.ts`** ‚Äî converts a numeric amount to Thai words (e.g. for billing printouts). Use the exported function rather than reimplementing.

### PWA & Offline

`public/sw.js` caches the app shell and falls back to `/offline` on network failure. `src/app/manifest.ts` defines PWA metadata. `PwaProvider` registers the service worker in production only. Do not modify `sw.js` without testing offline behavior.

## Coding Conventions

- **Style**: 2-space indentation, double quotes, semicolons, path alias `@/*` ‚Üí `./src/*`
- **Naming**: PascalCase for React components (`CustomerForm.tsx`), camelCase/descriptive lowercase for modules (`authorization.ts`)
- **Icons**: `lucide-react` only
- **Security**: Review `src/proxy.ts` and `src/lib/supabase/*` carefully when touching auth, sessions, or middleware
