# แนวทางขั้นตอนการจัดส่ง (Delivery Workflow Design)

## หลักการ: Order-to-Cash แบบเรียบง่าย

มาตรฐานสากลสำหรับธุรกิจจัดจำหน่ายขนาดเล็ก-กลาง (distributor/wholesaler) มีขั้นตอนหลัก 4 ช่วง:

```
รับออเดอร์ → ยืนยัน → จัดส่ง → เก็บเงิน
(Order)      (Confirm)  (Deliver)  (Collect)
```

ไม่ควรมีหน้าซ้ำกัน แต่ละหน้าทำหน้าที่เดียว ปุ่มไม่เกิน 3 ปุ่มต่อหน้า

---

## ขั้นตอนที่แนะนำ (4 สถานะ)

### สถานะออเดอร์

| สถานะ | ความหมาย | ใครทำ |
|-------|----------|-------|
| `draft` | ลูกค้ากรอกผ่าน LINE ยังไม่ส่ง | ลูกค้า |
| `submitted` | ลูกค้าส่งออเดอร์มาแล้ว | ลูกค้า |
| `confirmed` | แอดมินยืนยันและออก DN แล้ว | แอดมิน |
| `cancelled` | ยกเลิก | แอดมิน |

> ไม่ต้องมีสถานะมากกว่านี้ — ระบบ SAP B1 ระดับ SME ก็ใช้แค่ 4 สถานะ

---

## หน้าจอที่ควรมี (แนะนำ)

### 1. หน้า Orders — "กระดาน" รับออเดอร์ (มีอยู่แล้ว `/orders`)

**ทำหน้าที่:** ดูออเดอร์ที่ลูกค้าส่งมาวันนี้ → ยืนยันทีละร้าน

**ปุ่มที่มี:**
- ✅ ยืนยัน (submitted → confirmed + สร้าง DN อัตโนมัติ)
- ❌ ยกเลิก
- 👁 ดูรายละเอียด

**ไม่ควรมี:** ฟอร์มแก้ไขออเดอร์ในหน้านี้ (ทำให้ซับซ้อน)

---

### 2. หน้า Delivery — "ใบงานวันนี้" (`/delivery`) ← **ควรเพิ่ม**

**ทำหน้าที่:** ดู DN ที่ยืนยันแล้วทั้งหมดวันนี้ → พิมพ์ → ทำเครื่องหมายว่าส่งแล้ว

**แสดงข้อมูล:**
- รายชื่อร้านค้าที่ต้องส่งวันนี้ เรียงตามรถ/เส้นทาง
- สถานะ: รอส่ง / ส่งแล้ว / ปัญหา
- ปุ่ม "พิมพ์ทั้งหมด" — พิมพ์ DN ทุกใบในรอบวันนี้ต่อกัน

**ปุ่มที่มี:**
- 🖨 พิมพ์ใบส่งของ
- ✅ ส่งแล้ว
- ⚠️ มีปัญหา (บันทึกหมายเหตุ)

**ทำไมต้องแยกจาก Orders:**
- หน้า Orders ยุ่งกับการ "ตัดสินใจ" (ยืนยัน/ปฏิเสธ)
- หน้า Delivery ยุ่งกับการ "ปฏิบัติ" (พิมพ์/ส่ง)
- คนขับรถไม่ต้องเห็นออเดอร์ดิบ

---

### 3. หน้า Billing — "เก็บเงิน" (`/billing`) ← ถ้าต้องการในอนาคต

**ทำหน้าที่:** ออกใบวางบิลจาก DN ที่ส่งแล้ว

ปกติใช้ billing cycle ตาม `customers.billing_day` — ออกวางบิลรวมทุกวันที่ตรงกัน

---

## แผนผังขั้นตอน (Flow)

```
ลูกค้า LINE
    │
    ▼
[/order]  ลูกค้ากรอก + ส่งออเดอร์
    │
    ▼  (submitted)
[/orders]  แอดมินเช็ค → กด "ยืนยัน"
    │
    ▼  (confirmed + DN created)
[/delivery]  พนักงานดูใบงาน → พิมพ์ DN → ออกจัดส่ง
    │
    ▼  (delivered)
[/billing]  ออกวางบิลรวม (ตาม billing_day)
    │
    ▼  (paid)
จบ ✓
```

---

## หลัก UX สำหรับผู้ใช้ที่ไม่ถนัดเทคโนโลยี

1. **"หน้าเดียว = งานเดียว"** — ไม่ต้องคิดว่าอยู่หน้าไหน กด Back แล้วเริ่มใหม่ได้เสมอ
2. **ปุ่มใหญ่ สีต่างกัน** — ยืนยัน = เขียว, ยกเลิก = แดง, พิมพ์ = น้ำเงิน
3. **ไม่มี Modal ซ้อน** — ยืนยันก็ยืนยันเลย ไม่มี "คุณแน่ใจไหม?" หลายชั้น
4. **แสดงผลทันที** — กด ยืนยัน แล้วเห็นเปลี่ยนสถานะทันที ไม่รอโหลด
5. **วันที่ขึ้นต้นเสมอ** — ทุกหน้าแสดง "วันที่ X" ชัดเจน ไม่งง
6. **ไม่ต้องจำอะไร** — ชื่อร้าน รหัส ที่อยู่ ระบบแสดงให้ครบ

---

## สิ่งที่ไม่ควรทำ (Anti-patterns)

| ❌ อย่าทำ | ✅ ทำแบบนี้แทน |
|----------|--------------|
| หน้าเดียวทำทุกอย่าง | แยกหน้าตามบทบาท |
| ปุ่มเยอะ คำย่อ | ปุ่มน้อย ข้อความเต็ม |
| แก้ออเดอร์ที่ confirmed แล้ว | ยกเลิก + สร้างใหม่ |
| Export Excel ทุกอย่าง | พิมพ์ตรงจากระบบ |
| Dashboard กราฟเยอะ | ตาราง plain รายวัน |

## สรุปสำหรับ TYNoodle

**ระยะสั้น (ทำได้เลย):**
- หน้า `/orders` ที่มีอยู่ → ใช้สำหรับรับ + ยืนยันออเดอร์
- เพิ่มปุ่ม "พิมพ์ทั้งหมด" ใน `/orders` หรือ `/delivery` สำหรับพิมพ์ DN รวม

**ระยะกลาง:**
- สร้างหน้า `/delivery` — ใบงานวันนี้ + track สถานะ "ส่งแล้ว"

**ระยะยาว:**
- `/billing` — วางบิลรวมตาม `billing_day`
- รายงานสรุปรายเดือน
